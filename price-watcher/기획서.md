# 가격/재고 감시형 서비스 — 개발 기획 자료

## 1. 서비스 개요

**핵심 가치 제안**: 사용자가 24시간 붙어있지 않아도, 등록해 둔 조건(가격 하한선, 특정 키워드, 재고 부활)이 충족되는 순간 즉시 알림을 받아 기회를 놓치지 않는다.

**분류**: 데이터 중개형 / LLM 미사용 / 규칙 기반 자동화
**핵심 원칙**: 페인킬러 포지셔닝 (있어도 그만이 아니라, 놓치면 손해인 것)

---

## 2. MVP 범위 정의

| 항목 | MVP 결정 |
|---|---|
| 감시 대상 | 중고거래 플랫폼 1곳으로 시작 (예: 당근마켓 또는 번개장터) |
| 감시 조건 | 키워드 + 가격 상한선 (2가지 조합만) |
| 알림 방식 | 푸시 알림(PWA) 또는 텔레그램 봇 중 택1 |
| 조건 등록 개수 | 무료 1~2개 제한 |
| 크롤링 주기 | 초기 5분 간격 (실시간 아님, 검증 단계에서는 충분) |

> ⚠️ 처음부터 여러 플랫폼·복잡한 조건을 지원하려 하면 개발 범위가 커집니다. 플랫폼 1곳 + 조건 2종으로 먼저 검증하는 것을 권장합니다.

---

## 3. 기술 스택 후보

- **크롤링**: Python (requests + BeautifulSoup) 또는 Playwright (로그인/캡차 필요 시)
- **스케줄링**: cron 또는 APScheduler
- **DB**: SQLite(MVP) → PostgreSQL(스케일 시)
- **알림**: 텔레그램 봇 API(가장 빠른 구현) 또는 웹푸시(PWA)
- **프론트**: PWA (조건 등록 UI만 최소 구현)

---

## 4. 개발 전 확인이 필요한 핵심 질문 (법적/기술적 리스크)

1. **대상 플랫폼이 로그인 없이 접근 가능한 공개 데이터인가?**
   - 로그인 불필요 → 난이도·리스크 낮음
   - 로그인/캡차 필요 → Playwright 등 헤드리스 브라우저 필요, 계정 차단 리스크 있음
2. **크롤링 대상 플랫폼의 이용약관(ToS)에 크롤링 금지 조항이 있는가?**
   - 사업화 전 반드시 확인 필요 (개인 프로젝트와 상업 서비스는 리스크 수준이 다름)
3. **어떤 조건을 상진님 본인이 실제로 써봤을 때 가장 아쉬웠는가?**
   - 첫 헤비유저 = 본인이 되는 것이 검증 속도를 가장 높임

---

## 5. 수익화 모델 (재정리)

1. **프리미엄 알림 슬롯**: 무료 1~2개 조건 / 유료 무제한 + 알림 속도 우선
2. **성공 보수형(Pay-per-catch)**: 실제 매칭 성공 시에만 소액 과금
3. **B2B 수요 데이터 판매**: "이 조건을 감시하는 유저 수"를 익명화하여 판매자에게 제공 (중장기)
